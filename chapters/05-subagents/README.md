# 第5章: サブエージェント

> Status: 完了

## この章で学ぶこと

- サブエージェントとは「**自分のコンテキストを持つ専門作業者**」であること
- なぜメイン会話が汚れるのを防げるのか（並列・コスト・別視点の利点）
- `.claude/agents/*.md` のフロントマター（`name`・`description`・`tools`・`model`）の基本
- コマンド／スキルとの**使い分け**
- 実例: **MAGI**（3賢者の多数決）を「なぜこの形にしたか」で読む

## 前提

- [第4章: コマンド & スキル](../04-commands-skills/README.md)。拡張点を「ファイルで作る」感覚があること。

## 本文

### 1. サブエージェントは「もう1人の自分」

これまでの拡張点（コマンド・スキル）は、**同じ頭**が手順をこなすものでした。サブエージェントは違います。<mark>**別のコンテキスト（別の机）を持った、もう1人の Claude**</mark>に作業を任せます。

たとえば「このコードベースのどこにログイン処理があるか調べて」と頼むと、調査の過程で大量のファイルを読みます。それを**メインの会話**でやると、机（コンテキスト）が検索結果で埋まってしまう。そこで<mark>**調査専門のサブエージェントに丸投げし、結果の要点だけ受け取る**</mark>。メインの机はきれいなままです。

しょうへいさんの **MAGI**（3賢者にコードレビューさせて多数決）は、このサブエージェントの代表例です。

### 2. なぜ効くのか — 4つの利点

| 利点 | 中身 |
|---|---|
| **コンテキストを汚さない** | 大量の調査・出力を別の机でやり、要点だけ持ち帰る |
| **役割を固定できる** | 「レビュー専門」「調査専門」と人格を分けられる |
| **並列で走らせられる** | 複数のサブエージェントを同時に動かせる |
| **別視点で検証できる** | 同じ対象を別の観点で見せ、**多数決**で事故を減らす |

最後の「別視点での検証」が、MAGI の核心です。

### 3. 作り方 — `.claude/agents/` に定義する

サブエージェントも Markdown ファイルで作ります。

| 置き場所（Windows） | 効く範囲 |
|---|---|
| `<プロジェクト>\.claude\agents\<名前>.md` | そのプロジェクト |
| `C:\Users\zereh\.claude\agents\<名前>.md` | 全プロジェクト |

フロントマターで「どんな役割か・何のツールを使えるか・どのモデルで動くか」を決め、本文にそのエージェントへの指示（役割・方針）を書きます。

| 項目 | 意味 |
|---|---|
| `name` | エージェント名（必須） |
| `description` | いつ呼ぶか。これを手がかりに自動委譲される（必須） |
| `tools` | 使えるツールを限定（例: 読み取りだけ） |
| `model` | 使うモデル（`haiku` で軽く速く、`opus` でじっくり等） |

#### 最小の動く例：読み取り専用の調査エージェント

`<プロジェクト>\.claude\agents\researcher.md`:

```markdown
---
name: researcher
description: |
  コードベースを調べて要点だけ報告する読み取り専用エージェント。
  「どこにあるか調べて」「全体像を把握して」のような調査に使う。
tools: Read, Grep, Glob
model: haiku
---

あなたはコードベースの調査担当です。
- ファイルを読んで事実を確認し、要点だけ簡潔に報告する
- コードは変更しない（読み取り専用）
- 見つけた場所は ファイルパス:行番号 の形で示す
```

`tools` を `Read, Grep, Glob` に絞っているので、<mark>**このエージェントは間違ってもファイルを書き換えられません**</mark>。`model: haiku` で軽く速く。調査のような「数えて読むだけ」の作業にぴったりです。

> [!CAUTION]
> `tools` を**指定すると、その許可リストだけ**になります（メインで使えていた他のツールは引き継がれません）。「読み取りだけさせたい」ときには好都合ですが、「書けない」と気づかずハマることもあるので意図して絞ります。フロントマターには他にも項目がありますが、版で変わるため `name`/`description`/`tools`/`model` を主軸にし、応用は公式の *Subagents* ページで確認してください。

### 4. コマンド／スキルとの使い分け

| やりたいこと | 使うもの |
|---|---|
| 自分の決まった手順をまとめたい | コマンド／スキル（第4章） |
| 重い調査を別の机でやらせたい | **サブエージェント** |
| 別の観点で検証・多数決したい | **サブエージェント** |
| 複数を並列で走らせたい | **サブエージェント** |

ひとことで言うと、<mark>**「手順を固めたい → スキル」「頭を分けたい → サブエージェント」**</mark>。

### 5. 実例：MAGI は「うのみにしない」を仕組みにした

しょうへいさんの **MAGI** は、エヴァンゲリオンの MAGI システムになぞらえた**3賢者によるコードレビュー**です。3体のサブエージェントが、それぞれ**別の観点**で同じコードを見ます。

| 賢者 | 観点 | 使うツール |
|---|---|---|
| **MELCHIOR**（第一賢者） | 設計・アーキテクチャ | Read, Grep, Glob, Bash |
| **BALTHASAR**（第二賢者） | 保守性・可読性 | Read, Grep, Glob, Bash |
| **CASPER**（第三賢者） | セキュリティ・堅牢性 | Read, Grep, Glob, Bash |

そして3人の判定を**多数決**で「承認／否決」します。なぜこの形なのか。

- <mark>**「1回の出力をうのみにしない」**</mark>: LLM は間違いも自信満々に言う（第0章・旧 Ch1）。1人のレビューより、3人に別々に見せて多数決を取るほうが事故りにくい
- <mark>**観点を分けて網羅する**</mark>: 設計・可読性・セキュリティは見るポイントが違う。1人に全部やらせるより、観点ごとに専門家を立てるほうが漏れにくい
- **全員が読み取り中心（`Read, Grep, Glob, Bash`）**: レビューは「読んで判定する」仕事。書き換えツールを持たせない。第3章の権限の発想（読むのは自由、変えるのは慎重に）がここにも効いている

MAGI は、<mark>**LLM のクセ（間違いも堂々と言う）を、人数と観点の分担で補う**</mark>設計の好例です。サブエージェントの「別視点で検証できる」利点を、そのまま形にしています。

## つまずきポイント

> [!CAUTION]
> サブエージェントでハマりがちな点。

- **`tools` 指定で継承が消える**: 指定したツールだけになる。「なぜか書き込めない」は tools の絞りすぎを疑う
- **`description` が弱いと委譲されない**: 「いつ呼ぶか」を具体的に。曖昧だとメインが自分で抱え込む
- **コンテキストは引き継がれない**: サブエージェントは**別の机**。直前の会話の流れを前提にした指示は伝わらない。必要な情報は依頼文に書く
- **何でもサブエージェント化しない**: 軽い作業まで分けると、かえって往復が増える。重い・独立・別視点が要る作業に絞る
- **エンコーディング**: 定義ファイルは UTF-8（BOM なし）

## 重要な単語まとめ

| 用語 | 一言で言うと | 身近な実例 |
|---|---|---|
| **サブエージェント** | 別のコンテキストを持つもう1人の Claude | MAGI の3賢者 |
| **自動委譲** | description を手がかりにメインが作業を任せること | 「調べて」で調査エージェントが動く |
| **`tools`（ツール制限）** | そのエージェントが使えるツールを絞る | 読み取り専用（Read, Grep, Glob） |
| **`model`** | エージェントが使うモデル | 調査は haiku で軽く速く |
| **多数決レビュー** | 別観点の複数エージェントで判定し事故を減らす | MAGI の承認／否決 |
