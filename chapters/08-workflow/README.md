## 本文

### 1. 6つの拡張点は「組み合わせて」使う

第0章で地図を広げ、各章で1つずつ拡張点を見てきました。実運用では、これらは**単独ではなく組み合わさって**働きます。しょうへいさんの1日を例にすると、全部が連動しているのが分かります。

| 場面 | 効いている拡張点 |
|---|---|
| セッション開始時、推測せず確認してくる | **CLAUDE.md**（推測禁止ルール・グローバル） |
| 「メモして」で Vault に保存される | **スキル**（/save） |
| コードレビューを多数決でかける | **サブエージェント**（MAGI） |
| 学びを Zenn ログに追記して push | **スキル**（/zenn-log） |
| Slack 送信の前に内容を提示される | **権限**（送信系の事前確認） |
| Notion にタスクが登録される | **MCP**（Notion 連携） |

<mark>**「やりたいこと」ごとに最適な拡張点を選び、足りなければ作る**</mark>。これが「使いこなしている」状態の正体です。

### 2. 複数プロジェクトを管理する

プロジェクトが増えても、原則はシンプルです。

- <mark>**自分の働き方の基本はグローバル CLAUDE.md**</mark>（`~/.claude/`）に一度書く。推測禁止・Windows 注意・送信系確認など、どの案件でも守りたいこと
- **案件ごとの約束はプロジェクト CLAUDE.md**に書く。命名規則・進捗・その案件固有のルール
- スキルやサブエージェントも、**全案件で使うものはユーザースコープ**、**その案件専用はプロジェクトスコープ**に置き分ける

こうすると、新しいプロジェクトを始めても**「自分の基本装備」は最初から効いている**。毎回ゼロから設定し直す必要がありません。

### 3. Git との付き合い方 — 任せる線・自分で持つ線

何を Claude に任せ、何を自分の手元に残すか。実用的な目安です。

| Claude に任せてよい | 自分で判断・確認したい |
|---|---|
| 変更内容の確認・差分の説明 | **何をコミットするか**の最終判断 |
| コミットメッセージの下書き | `main` への直接コミット（避ける設定に） |
| 定型的なコミット作業（/commit） | **push**（外に出るので ask か手動） |

第1章の **plan モード**（動く前に計画を見る）と、第3章の **権限**（push は ask）を組み合わせると、<mark>**「Git は任せるが、外に出る一歩手前で必ず一拍置く」**</mark>運用ができます。

### 4. コスト最小で回す原則

しょうへいさんが重視する「コストゼロ／最小コスト」を、ここまでの章の道具で実現します。

- **CLAUDE.md を厳選**（150行ルール・第2章）: 毎回読まれるので、長いほど毎回のコスト。要点だけ
- **コンテキストを管理**（第1章）: 話題が変わったら `/clear`、長くなったら `/compact`。机を軽く保つ
- **軽いモデルを使い分け**（第5章）: 調査のような単純作業は `haiku` のサブエージェントへ
- **重い作業はサブエージェントへ分離**（第5章）: メインの机を検索結果で埋めない
- **過剰実装しない**: 1つの skill / hook に詰め込みすぎない。1テーマ1つ

> [!CAUTION]
> <mark>**「便利だから全部入れる」は、かえってコストと混乱を招く**</mark>。MCP をつなぎすぎればツールが増えてコンテキストを圧迫し、フックを盛れば動作が重くなる。**必要になってから足す**のが、結局いちばん安くて速い。

### 5. 「Claude Code 単体で足りる／追加は補完」の判断軸

拡張点を学ぶと、何でも作りたくなります。でも判断軸はシンプルです。

```
まず Claude Code 単体（素の対話）で試す
   └─ 毎回同じ前提を言っている        → CLAUDE.md に逃がす（第2章）
   └─ 同じ手順を繰り返している        → コマンド／スキルにする（第4章）
   └─ 重い・別視点が要る              → サブエージェント（第5章）
   └─ 「○○したら必ず△△」が欲しい     → フック（第6章）
   └─ 外部サービスを触りたい          → MCP（第7章）
   └─ 自動実行が怖い                  → 権限で絞る（第3章）
```

<mark>**「不便を感じてから、その不便に対応する拡張点を足す」**</mark>。先回りして作り込まない。これが過剰実装を避ける一番の方法です。

### 6. 完璧マスターのロードマップ総括

このハンドブックでたどった道のりを、一枚に畳みます。

| 段階 | 章 | 身につけたこと | 到達点 |
|---|---|---|---|
| **導入** | 第0章 | 6つの拡張点の地図 | 「どこに何があるか」が分かる |
| **初級** | 第1〜3章 | 基本操作・モード／CLAUDE.md／権限 | **安全に基本運用できる** |
| **中級** | 第4〜6章 | コマンド・スキル／サブエージェント／フック | **自動化を自分で作れる** |
| **上級** | 第7〜8章 | MCP／日常への統合 | **外部連携し、毎日に組み込める** |

到達点はシンプルです。<mark>**「Claude Code なしだと、自分の開発が遅く感じる」**</mark>。前提は CLAUDE.md が覚え、繰り返しはコマンドが引き受け、レビューは MAGI がかけ、保存は /save が片づけ、外部連携は MCP が運ぶ ── 手作業が仕組みに置き換わった状態。それが「完璧に使いこなす」ということです。

### 7. これで終わりではなく、ここから

Claude Code は**バージョンで進化し続けます**。本書が教えたのは「仕組みの理解」── これは古びません。一方、コマンド名・フラグ・フックのイベントといった**操作の細部は更新されます**。

だから運用のコツは1つ。<mark>**仕組みで理解し、細部は公式ドキュメントで確認する**</mark>。新機能が出たら、6つの拡張点のどれに当たるかを地図（第0章）で位置づければ、すぐに使いこなせます。

そして、このハンドブック自体が良い実践例です。`extras/qa-log.md` に「わからない」を記録し、たまったら補強ページにする。**自分の学びを仕組みに変えていく** ── それこそが、ここまで学んだことの総仕上げです。

## つまずきポイント

> [!CAUTION]
> 日常運用でハマりがちな点。

- **全部入れようとする**: 拡張点を覚えると作りたくなるが、不便を感じてから足すのが安い
- **CLAUDE.md の肥大**: 運用しているうちに膨らむ。定期的に 150行ルールで棚卸し
- **自動化しすぎて把握できない**: フック・権限・スキルが増えると「何が自動で動くか」を見失う。`/help`・設定ファイルで定期的に棚卸し
- **細部の暗記に頼る**: コマンド名を覚え込むより、仕組みで理解して公式で引く癖を

## おわりに

第0章で広げた地図を、ここまでで全部歩きました。CLAUDE.md・コマンド／スキル・サブエージェント・フック・MCP・権限 ── 6つの拡張点を、初級から上級まで実用本位でたどってきました。

あとは、<mark>**日々の「不便」を見つけて、対応する拡張点で潰していくだけ**</mark>。それを続ければ、Claude Code はあなた専用の相棒に育ちます。

## 重要な単語まとめ

| 用語 | 一言で言うと | 身近な実例 |
|---|---|---|
| **拡張点の組み合わせ** | 6つを連動させて使うこと | /save＋Notion MCP＋送信前確認 |
| **グローバル／プロジェクト分離** | 基本装備は全案件、約束は案件ごと | 推測禁止はグローバル、章ルールはプロジェクト |
| **任せる線／持つ線** | Git で何を委ね、何を自分で確認するか | 下書きは任せ、push は自分 |
| **必要になってから足す** | 過剰実装を避ける原則 | 不便を感じた拡張点だけ作る |
| **仕組みで理解し細部は公式で** | 古びない学び方 | 新機能を地図で位置づける |
