# 第13章: 検証駆動の開発ループ

> Status: 完了
> 発展（応用）コース。「できました」を、証拠で確かめてから信じる

## この章で学ぶこと

- LLM は「完了」も自信満々に言う ── だから**証拠で確認する**
- **検証駆動ループ**: 計画 → 実装 → テスト → 結果確認 → 修正
- 期待を先に決める（テスト・受け入れ基準を先に書く）
- plan モード・フックと組み合わせて自動で回す
- 「完了」の定義を先に決めておく

## 本文

### 1.「できました」をそのまま信じない

第0章で見たとおり、LLM は<mark>**間違いも「完了」も、同じ自信ありげな口調で言います**</mark>。「修正しました」「テストは通ります」と言われても、それは「それっぽい続き」かもしれない。

だから実務の鉄則は1つ。<mark>**主張ではなく、証拠で確認する**</mark>。「直した」なら実際にテストを走らせて緑を見る。「動く」なら実際に動かす。Claude Code には `verification-before-completion`（完了を主張する前に検証する）という考え方の skill もあるほど、ここは重要です。

### 2. 検証駆動ループ

そこで、作業を「ループ」として回します。

```
① 計画する（plan モード：何をどうするか先に見る）
        ↓
② 実装する
        ↓
③ テスト・実行する（証拠を出す）
        ↓
④ 結果を確認する  ──失敗──→ ②へ戻る
        ↓ 成功
⑤ 完了
```

ポイントは**③と④を飛ばさない**こと。実装したら必ず動かし、結果を自分の目で見る。失敗なら②に戻る。<mark>**「テストを実行して、その出力を見るまで完了と言わない」**</mark>。これが検証駆動です。

### 3. 期待を先に決める（テストファースト）

もう一歩進めると、**実装の前に「正解」を決めて**おけます。

- 「この関数は、入力 X に対して Y を返すべき」とテストを先に書く
- それから実装する
- テストが緑になったら完成

<mark>**ゴール（期待する結果）を先に固定する**</mark>と、Claude が「それっぽいけど違うもの」を作りにくくなります。第0章で学んだ「曖昧さ＝推測の余白」を、テストで先に埋めるわけです。Claude Code には `test-driven-development` の skill もあります。

### 4. plan モード・フックと組み合わせる

検証駆動は、これまでの章の道具で自動化できます。

- **plan モード（第1章）**: ①の計画を、動く前に見せてもらう
- **フック（第6・12章）**: `PostToolUse` で編集のたびに自動でテストを走らせれば、③④が自動で回る

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "npm test" }
        ]
      }
    ]
  }
}
```

> [!CAUTION]
> 編集のたびに**重いテスト全実行**を挟むと、待ち時間で作業が止まります。<mark>**軽い検証（関連テストだけ・lint だけ）を自動に、重い検証は節目で手動に**</mark>。第12章の「フックは軽く保つ」と同じ判断です。

### 5.「完了」の定義を先に決める

検証する前に、<mark>**「何をもって完了とするか」を決めておく**</mark>と、ループの出口がはっきりします。

- 「テストが全部緑」「lint エラーなし」「実際に画面で動く」など、**確認できる条件**で書く
- CLAUDE.md に「完了の条件」を書いておけば、毎回ぶれない（第2章）

> [!CAUTION]
> <mark>**「たぶん大丈夫」で完了にしない**</mark>。確認していない箇所は「未確認」と正直に言うのが、結局いちばん速い。曖昧な完了は、後工程で必ず手戻りになります（しょうへいさんの「推測禁止」と同じ精神）。

### 6. MAGI も検証の一形態

第5・10章の MAGI（多数決レビュー）も、広い意味で検証です。実装を「3つの観点でレビューして、多数決で承認する」。<mark>**1人の「できました」を、複数の目で確かめる**</mark>。検証駆動ループの「④結果確認」を、人数と観点で強化したものと言えます。

## つまずきポイント

> [!CAUTION]
> 検証駆動でハマりがちな点。

- **検証なしで完了宣言**: 「直しました」を鵜呑みにする。必ずテスト・実行の証拠を見る
- **テストを実行しない**: 「通るはず」で済ませる。実際に走らせて出力を確認
- **完了条件が曖昧**: 「いい感じ」では出口が決まらない。確認できる条件で書く
- **重い検証を毎回**: フックでテスト全実行すると遅い。軽い検証を自動、重いのは節目で
- **Windows でのテスト実行**: パス・改行・エンコーディングでテストが落ちることがある。環境差を疑う

## 重要な単語まとめ

| 用語 | 一言で言うと | 身近な実例 |
|---|---|---|
| **検証駆動ループ** | 計画→実装→テスト→確認→修正を回す | 緑を見るまで完了と言わない |
| **証拠で確認** | 主張でなく実行結果で判断する | テスト出力・実際の動作 |
| **テストファースト** | 正解を先に決めてから実装 | 期待値を先に書く |
| **完了の定義** | 何をもって完了とするか先に決める | 「テスト全緑」など確認できる条件 |
| **未確認は未確認と言う** | 曖昧な完了宣言をしない | 推測禁止と同じ精神 |
