## 本文

### 1. 起動と基本の対話

Claude Code はターミナルで使います。作業したいフォルダに移動して `claude` と打つだけです。

```powershell
# PowerShell（Windows）
cd "C:\Users\zereh\OneDrive\デスクトップ\プロジェクト\AI study"
claude
```

起動したら、あとは日本語で話しかければOK。最初に覚える操作はこれだけです。

| 操作 | 何をする | いつ使う |
|---|---|---|
| `/help` | 使えるコマンド一覧を出す | 迷ったら最初に |
| `/clear` | 会話をリセットする | 話題が変わったとき |
| `/compact` | 会話を要約して圧縮する | 会話が長くなってきたとき |
| `!<コマンド>` | シェルコマンドを直接実行する | ログインなど自分で打ちたいとき |

`!` プレフィックスは地味に便利です。たとえば `gcloud auth login` のような**対話的なログイン**を自分で打ちたいとき、プロンプトに `!gcloud auth login` と入れると、このセッション内で実行され、結果がそのまま会話に取り込まれます。ただし**実行結果は会話に取り込まれる**ので、パスワードやトークンが画面に表示されるコマンドは避けるか、出力の扱いに注意してください。

> [!CAUTION]
> <mark>**コマンド名やキー操作は Claude Code のバージョンで変わります**</mark>。本書のコマンド（`/compact` など）が見当たらないときは、まず `/help` で今の版の一覧を確認してください。画面下部のステータス表示も手がかりになります。

### 2. 3つのモード — 「どこまで自分で動いていいか」の設定

Claude Code は、**どこまで勝手に動いてよいか**を切り替えられます。ここを理解していないと「気づいたらファイルを書き換えられていた」「逆に毎回確認されて進まない」が起きます。

| モード | ふるまい | 向いている場面 |
|---|---|---|
| **通常モード** | 編集やコマンド実行のたびに許可を求める | 普段づかい。安全 |
| **plan モード** | まず**計画だけ**を立てて見せ、承認するまで実行しない | 影響が大きい・複数ファイルをいじる作業 |
| **自動承認モード** | 編集などを自動で通す（いちいち聞かない） | 信頼できる繰り返し作業を一気に進めたいとき |

モードの切り替えは、多くの版で **`Shift+Tab` を押すたびに循環**します。今どのモードかは画面下部に表示されます。

<mark>**plan モードは「先に計画を見せて」と頼む安全装置**</mark>です。しょうへいさんの「推測で進めず、まず要件を構造化した Q&A で絞ってから実装を委ねる」習慣と、発想はまったく同じ。<mark>**動く前に計画を見て、ズレていたらその場で直す**</mark> ── これを仕組みにしたのが plan モードです。

> [!CAUTION]
> キーバインド（`Shift+Tab` 等）やモードの正式名称は版で変わります。<mark>**「計画してから動くモードがある」「自動で通すモードがある」**</mark>という**概念**だけ覚えておけば、操作の細部が変わっても応用が効きます。正確なキー操作は `/help` と画面表示で確認してください。

#### 最小の動く例：plan モードを1回試す

1. `claude` を起動する
2. `Shift+Tab` を押して、画面下部の表示が **plan モード**に変わるのを確認する
3. 「この README に目次を追加して」など小さな依頼をする
4. すぐ書き換えず、**計画（やる手順）が提示される**のを見る
5. 問題なければ承認、ズレていれば「ここは違う」と指摘してから承認する

この「**計画 → 承認 → 実行**」の一拍が、事故をいちばん減らします。

### 3. コンテキスト — 「忘れる相手」とどう付き合うか

**コンテキスト**とは、Claude が**いまの会話で参照している情報のまとまり**です。あなたの発言・Claude の返答・読んだファイル・コマンドの結果 ── これらが全部、限られた「机の広さ（コンテキストウィンドウ）」に積み上がっていきます。

机には上限があります。だから会話が長く続くと、こうなります。

- <mark>**前のほうの指示を忘れる**</mark>（古い情報が机からあふれる）
- **だんだん精度が落ちる**（情報が多すぎて要点がぼやける）
- **コストが増える**（毎回、机いっぱいの情報を読み直すため）

> [!CAUTION]
> これは Claude の不注意ではなく、**仕組み上の上限**です。スマホで何十枚もタブを開くと重くなるのと同じ。<mark>**長い1本の会話で全部やろうとしない**</mark>のが、いちばん効くコツです。

#### 対処：`/clear` と `/compact` の使い分け

| やりたいこと | 使うもの | 何が起きる |
|---|---|---|
| 話題が完全に変わった | `/clear` | 会話をまっさらにリセット（前の文脈は消える） |
| 同じ作業を続けたいが会話が長い | `/compact` | これまでの会話を**要約**して圧縮し、続きを続行 |

**使い分けの目安**:

- バグ修正が終わって、次は別機能 → **`/clear`**（引きずらない）
- 長い実装の途中で動作が鈍ってきた → **`/compact`**（要点を残して軽くする）

### 4. 「会話に残す」か「CLAUDE.md に逃がす」か

コンテキストは会話のたびに消えたり圧縮されたりします。だから<mark>**「毎回必ず守ってほしい前提」を会話の中だけに書くのは危険**</mark>です。圧縮で消えたり、`/clear` で吹き飛んだりするからです。

そこで登場するのが次章の **CLAUDE.md**。毎回自動で読まれるので、消えません。

| 情報の種類 | どこに置く |
|---|---|
| その場限りの指示（「この関数だけ直して」） | 会話に書く |
| 毎回守ってほしいルール（「推測禁止」「送信前に確認」） | **CLAUDE.md に書く**（第2章） |
| プロジェクトの前提（環境・命名規則・進捗） | **CLAUDE.md に書く**（第2章） |

この「**消える記憶（コンテキスト）／消えない記憶（CLAUDE.md）**」の対比が、第2章の入り口になります。

## つまずきポイント

> [!CAUTION]
> 土台でハマりがちな点。

- **`/clear` は戻せない**: 会話をリセットすると前の文脈は消えます。続きをやりたいだけなら `/clear` ではなく `/compact`
- **長文の貼り付けは机を圧迫する**: 巨大なログやファイルを丸ごと貼ると、コンテキストが一気に埋まって精度が落ちます。必要な部分だけ貼るか、ファイルとして読ませる
- **キーバインドは版で変わる**: `Shift+Tab` の循環順や表示は更新されることがあります。`/help` と画面下部の表示が正
- **「自動承認モード」は便利だが油断は禁物**: 一気に進む反面、確認の一拍がなくなります。何を自動で許すかは第3章（権限）とセットで考える

## 重要な単語まとめ

| 用語 | 一言で言うと | 身近な実例 |
|---|---|---|
| **plan モード** | 計画を見せてから動く安全モード | 実装前に Q&A で要件を絞る習慣の自動版 |
| **自動承認モード** | 編集などを確認なしで通すモード | 信頼できる繰り返し作業を一気に進める |
| **コンテキスト** | いまの会話で参照中の情報のまとまり。上限あり | 開きすぎたスマホのタブで重くなる感じ |
| **コンテキストウィンドウ** | コンテキストの上限（机の広さ） | 長い会話で前の指示を忘れる原因 |
| **/clear** | 会話をリセットする | 話題が変わったとき |
| **/compact** | 会話を要約して圧縮する | 長い作業の途中で軽くしたいとき |
| **`!` プレフィックス** | シェルコマンドを直接実行する | `!gcloud auth login` などの対話ログイン |
