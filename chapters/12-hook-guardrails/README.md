# 第12章: フック多層ガードレール

> Status: 完了
> 発展（応用）コース。第6章「1つのフック」を、破られにくい多層防御へ

## この章で学ぶこと

- 1枚の柵（単一フック）は破られる、という前提
- **多層防御**: `deny`（権限）＋フック（条件判定）＋人の確認＋plan モードを重ねる
- フックで**ログを残す**（あとで何が起きたか追える）
- フックで**通知する**（危険操作・完了を知らせる）
- CASPER 的な「フックを唯一の安全柵にしない」を実装で示す

## 本文

### 1. 1枚の柵は、いつか破られる

第6章で、危険コマンドを止める `block-rm.sh`（grep で `rm -rf` を検出）を作りました。同時に「これは簡易デモで、`rm -fr` や連続スペースで容易にすり抜ける」とも書きました。

ここが本章の出発点です。<mark>**どんな単一のガードも、必ず抜け道がある**</mark>。だから安全は「1つの完璧な壁」ではなく、**「穴の位置が違う複数の薄い壁を重ねる」**で作ります。1枚目の穴を2枚目が塞ぐ ── いわゆる多層防御（スイスチーズモデル）です。

### 2. 多層防御の重ね方

Claude Code には、安全装置になる層が複数あります。これらを**重ねて**使います。

| 層 | 役割 | 章 |
|---|---|---|
| **plan モード** | 動く前に計画を見せる | 第1章 |
| **権限 `deny`** | 危険操作を機械的に禁止 | 第3章 |
| **フック（PreToolUse）** | 条件を見て止める | 第6章 |
| **人の確認（運用ルール）** | 送信系は実行前に提示 | 第3章 |

たとえば「破壊的な操作」への防御は、こう重ねます。

1. **`deny`** で `Bash(rm -rf:*)` を機械的に禁止（第3章）
2. **フック**で「想定外の rm パターン」も条件判定で止める（第6章）
3. **運用ルール**で、外に出る操作は人が最終確認（第3章）

> [!CAUTION]
> <mark>**「フックがあるから安全」ではありません**</mark>。フックは層の1つ。単純な禁止は `deny` のほうが確実で、フックは「中身を見て条件分岐する」用途に向きます。<mark>**まず deny で塞ぎ、足りない判断をフックで足す**</mark>。順序を間違えないこと。

### 3. フックでログを残す

多層防御のもう1つの柱が**記録**です。`PostToolUse` や `Stop` で「何が実行されたか」をログに残すと、<mark>**あとから「いつ・何が起きたか」を追える**</mark>。自動化が増えるほど、この監査ログが効いてきます。

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash|Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "bash ./.claude/hooks/log-action.sh"
          }
        ]
      }
    ]
  }
}
```

`log-action.sh` は stdin の JSON（どのツールが何をしたか）を受け取り、ログファイルに追記するだけ。<mark>**止められなかった操作も、記録さえあれば後で気づける**</mark>。これが「壁をすり抜けた後の網」になります。

### 4. フックで通知する

危険操作や、長いタスクの完了を**知らせる**のもフックの役目です。`Stop`（応答終了）で完了通知、`PreToolUse` で危険操作の警告 ── を出せば、<mark>**「気づかないうちに進んでいた」を防げます**</mark>。第9章の自動化・定期実行と組み合わせると、特に効きます。

### 5. 組み合わせの例

実戦では、複数のフックを役割分担させます。

- **PreToolUse**（Bash）: 危険コマンドを条件判定でブロック
- **PostToolUse**（Edit/Write）: 自動整形＋操作をログ
- **Stop**: 完了を通知

これに `deny`（機械的禁止）と運用ルール（人の確認）を重ねれば、<mark>**「止める・記録する・知らせる」が多層で回る**</mark>。CASPER が指摘した「フックを唯一の安全柵にしない」を、そのまま形にしたものです。

> [!CAUTION]
> フックは**重ねるほど動作が重くなり**ます。毎回テスト全実行のような重い処理を何枚も挟むと、編集のたびに待たされる。<mark>**軽く保ち、重い検証は必要なときだけ**</mark>（第13章の検証ループと使い分ける）。

## つまずきポイント

> [!CAUTION]
> フック多層化でハマりがちな点。

- **単層依存**: 「フック1枚で守れる」と思い込む。`deny`・人の確認と重ねる
- **ログの肥大**: 全操作を記録するとログが膨れる。重要な操作に絞る・定期的に切る
- **フックが重い**: 多層化で動作が遅くなる。軽い処理に保つ
- **順序の誤り**: 単純禁止は `deny` が先。フックは条件分岐が要るときだけ
- **Windows のエンコーディング**: フックスクリプトは UTF-8（BOM なし）、日本語インラインコメントは避ける

## 重要な単語まとめ

| 用語 | 一言で言うと | 身近な実例 |
|---|---|---|
| **多層防御** | 穴の位置が違う薄い壁を重ねる | deny＋フック＋人の確認 |
| **スイスチーズモデル** | 1枚の穴を次の層が塞ぐ考え方 | 1つのガードに頼らない |
| **監査ログ** | 何が起きたかを記録する | PostToolUse で操作を追記 |
| **通知フック** | 危険・完了を知らせる | Stop で完了通知 |
| **フックを唯一の柵にしない** | フックは層の1つに過ぎない | CASPER の指摘 |
