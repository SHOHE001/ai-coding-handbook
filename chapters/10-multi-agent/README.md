## 本文

### 1. 「1人」から「チーム」へ

第5章では、1体のサブエージェント（調査役・レビュー役）を作りました。本章は、それを**複数同時に・連携させて**動かします。

身近な例はすでにあります。しょうへいさんの **MAGI** は、3体のサブエージェント（MELCHIOR / BALTHASAR / CASPER）に同じコードを別観点で見せ、**多数決**で判定する。これがマルチエージェント・オーケストレーションそのものです。<mark>**1人の判断より、役割を分けた複数の判断のほうが事故りにくい**</mark> ── これを仕組みにします。

### 2. 3つの協調パターン

複数エージェントの束ね方は、大きく3つです。

| パターン | 動き | 向いている作業 |
|---|---|---|
| **並列（手分け）** | 複数が**同時に別の作業** | 大きなコードベースを範囲ごとに調査 |
| **パイプライン（順送り）** | 前の結果を**次に渡す** | 調査 → 実装案 → レビュー の流れ |
| **多数決（多視点）** | **同じ対象を別観点**で見て集約 | コードレビュー（MAGI）・重要判断 |

<mark>**作業の性質でパターンを選ぶ**</mark>。手分けできるなら並列、段階があるならパイプライン、間違えたくない判断なら多数決です。

### 3. 束ねる側（オーケストレーター）の仕事

複数エージェントを動かすとき、**指揮者**が要ります。多くの場合、それはメインの Claude です。指揮者の仕事は3つ。

1. **分割** — 大きなタスクを、エージェントに任せられる単位に切る
2. **委譲** — 各エージェントに「役割・対象・出力形式」を渡して走らせる
3. **集約** — 返ってきた結果をまとめ、最終判断する（MAGI なら多数決）

> [!CAUTION]
> サブエージェントは**それぞれ独立した机（コンテキスト）**を持ちます（第5章）。だから<mark>**指揮者が渡し忘れた情報は、エージェントには見えません**</mark>。「役割・対象・期待する出力」を依頼文に明示するのが、オーケストレーションの肝です。

### 4. 最小の動く例：調査 → レビューのパイプライン

第5章で作った読み取り専用エージェントを2体用意し、メイン会話から順に使います。

`.claude\agents\researcher.md`（調査役）と `.claude\agents\reviewer.md`（レビュー役）を置いた上で：

```
（メイン会話で）
まず researcher で認証まわりの実装箇所を洗い出して、
その結果を reviewer に渡してセキュリティ観点でレビューさせて。
```

指揮者（メインの Claude）が、researcher の結果を reviewer に**順送り**します。これがパイプラインの最小形です。MAGI のように「同じ対象を3体に並列で見せて多数決」にすれば、多視点パターンになります。

> [!CAUTION]
> オーケストレーションの**正確な起動方法・並列の仕組みは版で変わります**（サブエージェントの呼び出し方やツール名など）。本書は「役割を分けた複数エージェントに、分割・委譲・集約で仕事をさせる」という**考え方**を主軸にします。具体的な実行手段は公式の *Subagents* ページで確認してください。

### 5. MAGI に学ぶ設計

MAGI が多数決を採るのは、第0章で見た「LLM は間違いも自信満々に言う」への対策でした。マルチエージェント設計の教訓が詰まっています。

- **観点を分ける**: 設計・可読性・セキュリティを別エージェントに。1体に全部やらせるより漏れにくい
- **読み取りに絞る**: レビュー役には `Read, Grep, Glob` だけ。<mark>**判断するエージェントに、書き換える力を持たせない**</mark>（第3章の権限思想）
- **集約ルールを決める**: 「承認2票以上で可決」のように、まとめ方を先に決めておく

### 6. いつ使う / 使わない

| 使う | 使わない |
|---|---|
| 手分けできる大きな調査 | 数秒で終わる軽い作業 |
| 段階のある作業（調査→実装→レビュー） | 1往復で済む単純な依頼 |
| 間違えたくない重要判断（多数決） | 結果が自明なこと |

> [!CAUTION]
> <mark>**複数エージェントは、その数だけトークンを使います**</mark>。並列で5体走らせれば、単純計算でコストも増える。「速さ・確実さ」と「コスト」のトレードオフを意識し、**必要なときだけチームを組む**。第14章（コンテキスト・コスト戦略）と合わせて考えてください。

## つまずきポイント

> [!CAUTION]
> マルチエージェントでハマりがちな点。

- **何でもチーム化する**: 軽い作業まで分けると、往復とコストばかり増える。重い・並列可能・多視点が要るときに絞る
- **コンテキストの渡し忘れ**: 各エージェントは独立した机。前提を依頼文に書かないと的外れな結果が返る
- **集約の設計漏れ**: 結果を「どうまとめて何を判断するか」を決めずに走らせると、出力が散らかる
- **コスト管理**: 並列ほどトークンが膨らむ。`model`（haiku 等）の使い分けで軽くする
- **エンコーディング**: 各エージェント定義は UTF-8（BOM なし）
