## 本文

### 1. 「決まった手順」をワンコマンドに

毎回同じことを長文で頼んでいませんか。「git の変更を見て、意味のある単位でコミットして、メッセージは日本語で…」。これを毎回打つのは無駄です。

<mark>**よくやる手順を1つの呼び出しにまとめる**</mark> ── それがスラッシュコマンドとスキルです。しょうへいさんの `/save`（メモを仕分けて保存）や `/zenn-log`（学びをログに追記）が、まさにこれ。「いつもの段取り」をコマンド化したものです。

ざっくり言うと、**コマンドもスキルも「あらかじめ用意したプロンプト（指示書）」**です。違いは規模と呼ばれ方で、まずはコマンドから見ます。

### 2. スラッシュコマンド — 定型プロンプトをファイルにする

スラッシュコマンドは、**Markdown ファイルを置くだけ**で作れます。

| 置き場所（Windows） | 効く範囲 |
|---|---|
| `<プロジェクト>\.claude\commands\<名前>.md` | そのプロジェクト |
| `C:\Users\zereh\.claude\commands\<名前>.md` | 全プロジェクト |

ファイル名がそのままコマンド名になります（`commit.md` → `/commit`）。中身は「やってほしいこと」を書いた指示書です。

#### 最小の動く例：`/commit` コマンド

`<プロジェクト>\.claude\commands\commit.md` を作ります。

```markdown
---
description: 変更内容を確認して適切な粒度でコミットする
argument-hint: "[補足メッセージ（任意）]"
---

git の変更状況を確認し、意味のある単位でコミットしてください。
- コミットメッセージは日本語で簡潔に
- 関係ない変更を1つのコミットに混ぜない

補足の指示: $ARGUMENTS
```

これで `/commit` と打てば指示書が実行されます。`/commit ログイン機能のUIを調整` のように引数を渡すと、その文字列が **`$ARGUMENTS`** に入って指示書に差し込まれます。

- フロントマター（上の `---` で囲む部分）の `description` は、一覧やヒントに使われます
- `argument-hint` は、引数の入れ方のヒント表示です

> [!CAUTION]
> フロントマターで指定できる項目は版で増減します。まずは **`description` と `$ARGUMENTS` の2つ**を押さえれば実用上は十分。応用（実行ツールの制限など）は公式の *Slash commands* ページで確認してください。

### 3. スキル — 手順＋付属ファイルをまとめた「能力」

スキルは、コマンドの**ちょっと本格版**です。`SKILL.md` を**専用フォルダ**に置き、必要なら**補助ファイル（スクリプトやテンプレート）も同じフォルダに同梱**できます。

```
.claude/skills/
└── greet/
    ├── SKILL.md          ← 本体（必須・ファイル名は大文字 SKILL.md）
    ├── scripts/          ← 任意：補助スクリプト
    └── templates/        ← 任意：テンプレート
```

`SKILL.md` の冒頭には、最低限 **`name` と `description`** を書きます。

#### 最小の動く例：`greet` スキル

`<プロジェクト>\.claude\skills\greet\SKILL.md`:

```markdown
---
name: greet
description: |
  このプロジェクトの状況を一言で説明する。
  「状況教えて」「greet」と言われたとき、または
  作業を始める前の現状確認に使う。
---

現在のフォルダの README とファイル構成をざっと見て、
1. このプロジェクトが何かを1〜2文で説明
2. 次にやるとよさそうなことを1つ提案
してください。簡潔に。
```

`/greet` で手動でも呼べますし、<mark>**description に合う状況で Claude が自動で呼ぶ**</mark>こともあります。

> [!CAUTION]
> フォルダ名と中の `SKILL.md`（**大文字固定**）の対応を間違えると認識されません。`skills\greet\SKILL.md` のように、**フォルダ名＝スキル名**、ファイル名は必ず `SKILL.md`。高度なフロントマター項目は版で変わるので、まずは `name` / `description` を主軸に。

### 4. コマンドとスキル、どう使い分ける？

迷うのが普通です。地続きなので、まずはこの目安で十分。

| こういうとき | おすすめ |
|---|---|
| 指示が短い定型文。自分で `/` を打って呼ぶ | **スラッシュコマンド** |
| 手順が長い／付属ファイルがある／状況に応じて自動でも呼ばれてほしい | **スキル** |

<mark>**「短い定型はコマンド、まとまった能力はスキル」**</mark>。迷ったらコマンドで始めて、育ってきたらスキルに引っ越す、で問題ありません。

### 5. `description` が運命を分ける

コマンドもスキルも、<mark>**`description` を手がかりに Claude が「これ使えそう」と判断**</mark>します。だから description の質が、自動発動の精度を決めます。

```
弱い description:
  「保存します」
  → いつ使うか不明。呼ばれない or 誤爆する

強い description:
  「メモ・アイディア・読書記録を保存したいとき。
   『メモして』『保存して』『これ書いておいて』『本読んだ』
   と言われたら使う。保存先の判断・リンク・タグ付けも行う」
  → トリガーが明確。狙ったときに呼ばれる
```

ポイントは<mark>**「いつ使うか」「どんな言葉で呼ばれたいか」を具体的に書く**</mark>こと。第0章で「自動で呼ばれる仕組みがある」と言ったのは、ここに効いてきます。

### 6. 実例：`/save` と `/zenn-log` は「なぜこの形か」

しょうへいさんが実際に作って運用しているスキルを読むと、設計意図が見えます。

**`/save`（内容を判定して Vault に保存）**

> 内容を判定して Obsidian Vault に保存し、関連ノートにリンク、タグ付与、Notion 並行保存まで自動で行う。

ここでの設計判断は <mark>**「保存先の判断という面倒な部分こそ、スキルに委ねる」**</mark>。ユーザーは内容を投げるだけ。場所選び・リンク張り・タグ付けをやらせない。だから description には「メモして」「保存して」「これ書いておいて」「本読んだ」といった**トリガー語を厚く列挙**して、思いついた瞬間どんな言い方でも発動するようにしてあります。これは「自動発動の精度を description で作り込む」の見本です。

**`/zenn-log`（学びをログに追記して git push）**

> Claude との会話で学んだことを Zenn の月別ログ記事に追記して git push する。スクラップっぽい運用。

こちらは**「定型手順の塊」**。月別ファイルを探す → 追記する → git に push する、という決まった段取りを毎回手で打たずに済ませる。手順がはっきりしていて付属の運用ルールがあるので、コマンドではなくスキル向き、という判断です。

両者を比べると、<mark>**「判断を委ねたいか（/save）」「手順を固めたいか（/zenn-log）」**</mark>でスキルの設計が変わるのが分かります。

## つまずきポイント

> [!CAUTION]
> コマンド・スキルでハマりがちな点。

- **description が弱くて呼ばれない**: 「いつ・どんな言葉で」使うかを具体的に。トリガー語を入れる
- **`SKILL.md` のファイル名・場所ミス**: 大文字 `SKILL.md`、フォルダ名＝スキル名。`commands` と `skills` の置き場所を取り違えない
- **`$ARGUMENTS` の入れ忘れ**: 引数を使うコマンドなのに本文に `$ARGUMENTS` がないと、渡した値が効かない
- **Windows のエンコーディング**: `SKILL.md` も `.md` コマンドも UTF-8（BOM なし）。実行する補助スクリプトに**日本語のインラインコメントを入れない**（コードページ文字化けでハマった実績あり）
- **高度なフロントマターを推測で使う**: 版で変わる項目を当て推量で書くと動かない。`name`/`description` を主軸にし、応用は公式で確認

## 重要な単語まとめ

| 用語 | 一言で言うと | 身近な実例 |
|---|---|---|
| **スラッシュコマンド** | 定型プロンプトを `.md` ファイルにしたもの | `/commit`・`/zenn-log` |
| **`$ARGUMENTS`** | コマンドに渡した引数が入る場所 | `/commit ログイン機能` の「ログイン機能」 |
| **スキル / SKILL.md** | 手順＋付属ファイルをまとめた能力 | `/save`（保存を全自動化） |
| **description** | Claude が「使うべき」と判断する手がかり | トリガー語を列挙して自動発動させる |
| **コマンド↔スキルの使い分け** | 短い定型はコマンド、まとまった能力はスキル | 短文は `/commit`、判断込みは `/save` |
