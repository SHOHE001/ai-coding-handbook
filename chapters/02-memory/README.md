# 第2章: プロジェクトの記憶 — CLAUDE.md とメモリ

> Status: 完了

## この章で学ぶこと

- CLAUDE.md が「プロジェクトの記憶」＝**毎回自動で読まれる常駐メモ**であること
- **3階層**の使い分け: プロジェクト直下 / ユーザーグローバル / サブフォルダ
- 何を書くと効くか: 運用ルール・進捗・禁止事項・前提
- 良い CLAUDE.md / 悪い CLAUDE.md の違い
- **150行ルール**: 常駐ぶん長いほど毎回コストが増える。厳選する
- `#` でのクイックメモ追記と `/memory` での編集
- 実例として、しょうへいさんが運用中の CLAUDE.md（推測禁止・送信系の事前確認・Windows 注意）を読む

## 前提

- [第1章: 土台](../01-basics/README.md)。「消える記憶（コンテキスト）／消えない記憶（CLAUDE.md）」の対比が頭にあること。

## 本文

### 1. CLAUDE.md は「毎回読まれる常駐メモ」

第1章で見たとおり、会話のコンテキストは圧縮されたり `/clear` で消えたりします。だから「毎回必ず守ってほしいこと」を会話に書くのは不安定です。

そこで **CLAUDE.md**。<mark>**セッションを始めるたびに、Claude Code が自動で最初に読むファイル**</mark>です。ここに書いたことは、毎回の会話に「前提」として効きます。

身近に言えば、CLAUDE.md は<mark>**新人に毎回渡す「業務マニュアル」**</mark>です。優秀だけど思い込みの激しい新人に、毎朝同じ説明をするのは無駄。一度マニュアルに書いておけば、毎回読んでから仕事を始めてくれる ── それが CLAUDE.md の役割です。

> [!CAUTION]
> CLAUDE.md は<mark>**「永続するシステムプロンプト」**</mark>のように働きます。会話で1回お願いするのと違い、**常に効き続ける**。だからこそ、何を書くか（書かないか）が効いてきます。

### 2. 3階層 — どこに置くと、どこまで効くか

CLAUDE.md は置き場所で「効く範囲」が変わります。

| 置き場所 | パス（Windows） | 効く範囲 | 用途 |
|---|---|---|---|
| **プロジェクト直下** | `<プロジェクト>\CLAUDE.md` | そのプロジェクトだけ | プロジェクト固有のルール・進捗。git で共有できる |
| **ユーザーグローバル** | `C:\Users\zereh\.claude\CLAUDE.md` | 全プロジェクト共通 | 自分の働き方の基本方針（推測禁止・Windows 注意など） |
| **サブフォルダ** | `<プロジェクト>\sub\CLAUDE.md` | そのフォルダ以下 | モノレポで部分ごとにルールを変えたいとき |

しょうへいさんの環境では、まさにこの2層が動いています。

- **グローバル**（`C:\Users\zereh\.claude\CLAUDE.md`）: 「推測禁止ルール」「送信系操作は事前確認」「Windows のエンコーディング注意」など、**全プロジェクトで守りたい自分の働き方**
- **プロジェクト直下**（このハンドブックの `CLAUDE.md`）: 「章の構成ルール」「3箇所同期」「正確性ポリシー」など、**この本だけのルール**

<mark>**「いつもの自分のクセはグローバル、その案件だけの約束はプロジェクト」**</mark>と覚えると迷いません。

> [!CAUTION]
> CLAUDE.md の細かい仕組み（個人用の上書きファイル、`@パス` での他ファイル取り込みなど）は版で増減します。まずは<mark>**「グローバル」と「プロジェクト直下」の2つ**</mark>を押さえれば実用上は足ります。応用は公式の *Memory* ページで確認してください。

### 3. 何を書くと効くか

効くのは、**Claude が推測で埋めてしまう余白**を先に埋める情報です。

| 書くと効く | 例 |
|---|---|
| **運用ルール・禁止事項** | 「不明点は推測せず質問する」「`main` に直接コミットしない」 |
| **前提・環境** | 「Windows + PowerShell」「ファイルは UTF-8（BOM なし）で書く」 |
| **進捗・状態** | 「いま第2章まで完了」「次は権限章」 |
| **命名・構成のルール** | 「章は `NN-テーマ/README.md`」「送信系は実行前に内容提示」 |

逆に、**書かないほうがいい**ものもあります。

- コードを読めば分かること（関数の中身・ファイル構成）→ Claude はファイルを読めます。書くと二重管理で古くなる
- その場限りの指示 → 会話で言えば十分
- パスワード・APIキーなどの秘密 → **git 共有される CLAUDE.md には絶対に書かない**

### 4. 良い CLAUDE.md / 悪い CLAUDE.md

```
悪い例:
  「いい感じにやってください。よろしく。」
  → 何を守ればいいか分からない。推測の余白が広いまま

良い例:
  「## ルール
   - 不明点は推測せず質問する
   - 送信系（メール・Slack）は実行前に内容を提示して承認を得る
   - ファイルは UTF-8（BOM なし）で保存する」
  → 守るべき線がはっきりしている
```

ポイントは<mark>**「具体的・簡潔・最新」**</mark>。曖昧だと効かず、長すぎると埋もれ、古いと害になります。

#### 最小の動く例：1分で作る CLAUDE.md

プロジェクト直下に `CLAUDE.md` を作り、これだけ書けば十分スタートできます。

```markdown
# CLAUDE.md

## このプロジェクトについて
個人用の〇〇を作る。Windows + PowerShell 環境。

## ルール
- 不明点は推測せず質問する
- 既存の動いているコードは、理由なく変えない
- ファイルは UTF-8（BOM なし）で保存する

## 進捗
- [ ] まだ何もしていない
```

> [!CAUTION]
> Windows でこのファイルを作るとき、PowerShell の `Out-File` / `Set-Content` は**デフォルトが UTF-16**です。必ず `-Encoding utf8` を付けるか、VS Code 等で「UTF-8（BOM なし）」を選んで保存してください。BOM 付きや UTF-16 だと、他のツールが読めずに文字化けします（しょうへいさんが何度もハマってきた点）。

### 5. 150行ルール — 長いほど毎回コストになる

CLAUDE.md は**毎セッション必ず読まれます**。つまり<mark>**長くなるほど、毎回のコストが増え、肝心のルールが埋もれる**</mark>。

しょうへいさんのグローバル CLAUDE.md には、まさにこの戒めが書かれています。

> 1ファイルが 150 行を超えたあたりから冗長とみなす。超えていたら「整理・分割を検討してください」と警告する。

実用上の目安:

- **150行を超えたら**、削るか、テーマごとに分割するかを検討する
- 「読めば分かること」は消す（コード構成など）
- 本当に毎回効かせたい**ルールと前提だけ**を残す

### 6. `#` クイックメモと `/memory`

毎回ファイルを開かなくても、会話の流れでメモを足せます（版により挙動は異なります）。

- プロンプトの先頭に **`#`** を付けて書くと、その内容を **CLAUDE.md に追記する候補**として扱ってくれる
  ```
  # このプロジェクトでは npm ではなく pnpm を使う
  ```
- **`/memory`** で、いまのメモリ（CLAUDE.md）を開いて確認・編集できる

「あ、これ毎回必要だ」と気づいた瞬間に `#` で書き留め、あとで `/memory` で整える ── という運用が軽くて続きます。

## つまずきポイント

> [!CAUTION]
> CLAUDE.md でハマりがちな点。

- **長すぎて埋もれる**: 全部書くと逆に効かない。150行を目安に厳選（後述の「150行ルール」）
- **古いまま放置**: 進捗や方針が変わったのに更新しないと、Claude が古い前提で動く。変えたら同時に直す
- **グローバルとプロジェクトの混同**: 全案件のクセをプロジェクトに書くと他で使えない。逆も同じ。層を意識する
- **秘密情報を書いてしまう**: git 共有される CLAUDE.md に API キー等を書かない。秘密は環境変数や別の仕組みへ
- **Windows のエンコーディング**: UTF-8（BOM なし）以外で保存すると文字化け・読み込み失敗
