## 本文

### 1. MCP は「外部サービスへの手足」

Claude Code は標準だと、**手元のファイルとコマンド**しか触れません。Notion にメモを書く、Slack を読む、カレンダーに予定を入れる ── こうした**外のサービス**を操作するには、手足を伸ばす仕組みが要ります。それが **MCP（Model Context Protocol）** です。

<mark>**MCP は「AI と外部サービスをつなぐ共通プラグ」**</mark>。一度つなぐと、その外部サービスの操作が Claude の**ツール**として使えるようになります。しょうへいさんの環境では、すでに **Notion・Slack・Gmail・Google Calendar・Google Drive・NotebookLM** などがこの仕組みでつながっています。

### 2. つなぎ方 — `claude mcp add`

MCP サーバーは、ターミナルの `claude mcp add` コマンドで追加します。種別は主に2つ。

| 種別 | 何か | 例 |
|---|---|---|
| **http** | リモートの URL につなぐ | クラウド上のサービス |
| **stdio** | 手元でサーバープログラムを起動してつなぐ | ローカルのツール |

```powershell
# http（リモートURL）につなぐ
claude mcp add --transport http example https://example.com/mcp

# stdio（手元でサーバーを起動）：npx で公式サンプルを起動する例
claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem C:\Users\zereh\work
```

つないだら、Claude Code の中で <mark>**`/mcp`**</mark> と打つと、接続状態の確認や（必要な場合の）ログイン認証ができます。

> [!CAUTION]
> `claude mcp add` のフラグ（`--transport` など）や認証の流れは版で変わります。<mark>**つなぎたいサービスの「公式 MCP 手順」に従う**</mark>のが確実です。本書はコマンドの形だけ示します。細部は公式の *MCP* ページと、各サービス提供元の案内で確認してください。

### 3. スコープ — どこまで使えるようにするか

CLAUDE.md や settings.json と同じく、MCP にも「効く範囲」があります。

| スコープ | 範囲 | git 共有 | 用途 |
|---|---|---|---|
| **local** | 自分・このプロジェクトだけ | ✗ | 個人用・トークンを含む接続 |
| **project** | このプロジェクト（`.mcp.json`） | ✓ | チームで共有したい接続 |
| **user** | 自分の全プロジェクト | ✗ | どこでも使う個人ツール |

```powershell
# 例：プロジェクト共有（.mcp.json に書かれ、git で共有できる）
claude mcp add --scope project example https://example.com/mcp
```

`--scope project` で追加すると `.mcp.json` に記録され、チームで同じ接続を共有できます。

### 4. 認証・秘密情報の扱い

外部サービスにつなぐ以上、**トークン（鍵）**を扱います。ここは第3章の「秘密を漏らさない」と直結します。

- <mark>**トークンを `.mcp.json` に直書きして git に上げない**</mark>。共有ファイルに鍵を書くのは事故のもと
- 環境変数や、各サービスのログイン認証（OAuth）を使う
- 個人用・トークン込みの接続は **local スコープ**（git 非共有）に置く

### 5. 第3章との合わせ技 — 送信系は二重に守る

MCP でつないだ操作も、**第3章の権限の対象**です。MCP ツールは多くの版で `mcp__<サーバー名>__<ツール名>` のような名前を持ち、`allow`/`deny`/`ask` で制御できます。

ここで効いてくるのが、しょうへいさんの<mark>**「送信系は事前確認」運用**</mark>です。

> Slack 送信・Gmail ドラフト・Calendar 作成・Notion コメントなど、**外部チャネルへ飛ぶ操作**は、allow に含まれていても**実行前に内容を提示して承認を得る**。

なぜこの形か。<mark>**読み取り（Slack を読む・カレンダーを見る）は取り返しがつくが、送信（メッセージを送る・予定を作る）は外の人に届いてしまい、取り返しがつかない**</mark>。だから：

- **読み取り系**（`slack_read_*`・検索など）→ 許可して快適に
- **送信系**（`slack_send_*`・`create-event` など）→ 許可していても**人の目を一度通す**

第3章で見た「機械的な allow ＋ 人の確認ルール」という<mark>**二重の安全装置**</mark>を、外部連携でこそ徹底する。MCP は便利なぶん「外に届く」操作を含むので、この線引きが一番効きます。

### 6. 実例：なぜ Notion・Slack・Gmail をこの形でつないでいるか

しょうへいさんの連携は、用途で役割が分かれています。

- **Notion**: タスク登録・メモ保存の保存先（`/save` や add-task の行き先）。書き込みも使うが、**通知が飛ぶコメント（`notion-create-comment`）は事前確認**
- **Slack**: 読み取り・検索は自由、**送信・予約投稿は事前確認**
- **Gmail / Calendar**: 検索・閲覧は許可、**ドラフト作成・予定作成（参加者へ通知）は事前確認**

共通する設計思想は、<mark>**「中に取り込む（読む・検索する）は気軽に、外へ出す（送る・作る・通知する）は慎重に」**</mark>。第3章・第6章で繰り返してきた線引きが、外部連携で完成形になります。

## つまずきポイント

> [!CAUTION]
> MCP でハマりがちな点。

- **接続できない**: stdio サーバーは起動コマンド（`npx ...` のパスや引数）が正しいか確認。`/mcp` で状態を見る
- **トークン漏れ**: `.mcp.json` に鍵を直書きして git に上げてしまう事故。秘密は local スコープ・環境変数へ
- **スコープの取り違え**: 個人トークン込みを project スコープにすると共有されてしまう。個人用は local
- **送信系の自動実行**: 「許可したつもり」が外部に通知を飛ばす。送信系は ask か運用ルールで人の確認を挟む
- **Windows のパス**: stdio サーバーに渡すパスのバックスラッシュ・スペースに注意。引数はクォートする
- **つなぎすぎ**: 多数の MCP を常時つなぐと、使えるツールが増えてコンテキストを圧迫する。必要なものだけ
