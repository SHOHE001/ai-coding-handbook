# 第11章: スキルを育てる

> Status: 完了
> 発展（応用）コース。第4章「スキルを作る」から、「スキルを育てて運用に乗せる」へ

## この章で学ぶこと

- スキルは作って終わりではなく、**使いながら育てる**もの
- 補助ファイル（スクリプト・テンプレート）を**同梱**して手順を外出しする
- 引数（`$ARGUMENTS`）で**汎用化**する
- 1スキルに詰め込みすぎず、**分割・連携**させる
- CLAUDE.md が手続きで肥大したら、**スキルに切り出す**
- 実例: `/save`・`/sort`・`/zenn-log` の育ち方

- 発展コースの各章は**独立した読み切り**です。

## 本文

### 1. スキルは「育てる」もの

第4章で作ったスキルは、最初は数行で十分でした。でも使ううちに「ここも自動化したい」「この定型文も入れたい」と要望が増えます。<mark>**スキルは、運用しながら少しずつ太らせていく**</mark>のが自然な姿です。

しょうへいさんの `/save` がいい例です。最初は「保存する」だけでも、今は「内容を判定 → 保存先決定 → リンク → タグ付け → Notion 並行保存」まで育っています。<mark>**最初から完璧を目指さず、不便を感じた部分を継ぎ足していく**</mark>。

### 2. 補助ファイルを同梱する

スキルが育つと、`SKILL.md` 本文だけでは窮屈になります。第4章で触れたとおり、スキルは**専用フォルダ**なので、補助ファイルを一緒に置けます。

```
.claude/skills/
└── save/
    ├── SKILL.md          ← 手順の本体
    ├── scripts/          ← 補助スクリプト（重い処理を外出し）
    │   └── classify.ps1
    └── templates/        ← 定型テンプレート
        └── note.md
```

<mark>**長い処理はスクリプトへ、繰り返す定型文はテンプレートへ**</mark>。`SKILL.md` は「何をするか」の見通しを保ち、細部は外のファイルに逃がす。こうすると、本文が読みやすいまま機能を足せます。

> [!CAUTION]
> 補助スクリプトを Windows で動かすとき、<mark>**スクリプトに日本語のインラインコメントを入れない**</mark>（コードページの文字化けで関数定義が壊れた実績あり）。ファイルは UTF-8（BOM なし）。スキルから呼ぶパスは、フォルダ内の相対パスで指定すると移動に強くなります。

### 3. 引数で汎用化する

「特定の1件」しか扱えないスキルは応用が利きません。`$ARGUMENTS`（第4章）で**外から対象を渡せる**ようにすると、一気に汎用になります。

```markdown
---
name: sort
description: 指定日のデイリーノートを仕分ける。「仕分けして」「整理して」で発動。引数で日付指定も可。
---

対象: $ARGUMENTS （未指定なら今日のノート）

つぶやきをセクション分けし、リンク・タグを付け、アイディアは独立ファイルにしてください。
```

`/sort` なら今日、`/sort 2026-05-07` なら指定日 ── 同じスキルが両方こなせます。しょうへいさんの `/sort` が、まさにこの「引数なし＝今日、日付指定で過去分」の形です。

### 4. 詰め込みすぎず、分割する

育てるうちに「1つのスキルに何でも入れたく」なりますが、これは罠です。

> [!CAUTION]
> <mark>**1スキル＝1テーマに保つ**</mark>。保存も仕分けもログ投稿も全部入りの巨大スキルは、description が曖昧になって**自動発動の精度が落ち**、保守も難しくなります。`/save`（保存）・`/sort`（仕分け）・`/zenn-log`（ログ投稿）のように、**役割でスキルを分ける**ほうが、それぞれ的確に呼ばれます。

### 5. スキルを連携させる

分けたスキルは、**つなげて**使えます。あるスキルの結果を、別のスキルや手順に渡す。たとえば「会話で学んだ → `/save` で保存 → 月末に `/zenn-log` でまとめて投稿」のように、小さなスキルの連なりで大きな運用が回ります。<mark>**小さく分けて、つないで使う**</mark>のが、育てたスキルの活かし方です。

### 6. CLAUDE.md が肥大したら、スキルへ逃がす

第2章で「CLAUDE.md は 150行を目安に厳選」と学びました。運用していると、CLAUDE.md に**手順（手続き）**が溜まってきます。「○○するときは、まず△△して、次に□□して…」のような段取りです。

これは<mark>**スキルに切り出すサイン**</mark>です。CLAUDE.md は「毎回読まれる記憶」。手順を書くと毎回コストがかかり、肝心のルールが埋もれます。手続きはスキル（必要なときだけ読まれる）へ移し、CLAUDE.md は「前提とルール」に絞る。これが第2章と第4章をつなぐ運用のコツです。

### 7. 実例：育ったスキルたち

しょうへいさんのスキル群は、それぞれ役割を持って分かれています。

- **`/save`**: 判断を委ねる系。保存先決定・リンク・タグ・Notion 並行まで育った
- **`/sort`**: 引数で汎用化（今日 / 日付指定 / 週次）
- **`/zenn-log`**: 定型手順の系。月別ログに追記して git push
- **`/log`**: セッションを振り返って Notion に記録

<mark>**「判断を委ねたい」「手順を固めたい」「引数で広げたい」**</mark> ── 育て方は目的で変わります。自分のスキルがどれを目指すのかを意識すると、迷わず太らせられます。

## つまずきポイント

> [!CAUTION]
> スキルを育てるときにハマりがちな点。

- **詰め込みすぎ**: 何でも入りスキルは description が曖昧になり呼ばれにくい。役割で分ける
- **description の更新忘れ**: 機能を足したのにトリガー語を増やさないと、新機能で呼ばれない
- **補助ファイルのパス**: 絶対パスで書くと移動に弱い。フォルダ内の相対パスで
- **CLAUDE.md に手順を溜める**: 手続きはスキルへ。CLAUDE.md は前提とルールに絞る
- **Windows のエンコーディング**: SKILL.md も補助スクリプトも UTF-8（BOM なし）、日本語インラインコメントは避ける
