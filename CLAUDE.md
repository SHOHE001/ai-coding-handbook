# CLAUDE.md — Claude Code 実戦ハンドブック

このファイルは、Claude Code がこのプロジェクトで作業するときの **運用ルール** と **進捗** をまとめたもの。新しいセッションを始めるとき、Claude Code はまずこの文書を読んでから作業を開始する。

進捗や方針が更新されたら、このファイルも更新する。

---

## プロジェクトについて

非エンジニアの個人開発者が、**Claude Code を「完璧に使いこなす」ための実用ハンドブック**。読み物ではなく、**自分で skill / コマンド / サブエージェント / フック / MCP を作って運用に乗せられる**状態を目標にする。

- 旧版（概念中心の読み物）から、**実用・段階的（初級→上級）** に全面作り替えた
- 旧版の原稿は `extras/archive/old-chapters/` に保全（Ch1 概念本文は資産として参照可）
- リポジトリ: public を想定。姉妹リポジトリ `web-basics-handbook` の作りを踏襲

---

## ユーザー（読者）の前提

- 非エンジニア、大学 4 年生。Claude Code は日常的に運用済み（CLAUDE.md・スラッシュコマンド・サブエージェント・MCP・skill を実践済み）
- **実行環境は Windows 11 + PowerShell / Git Bash 併用**。実用例は必ずこの環境で動く形で書く
- 「過剰実装は不要、シンプルに」。コストゼロ／最小コストの原則を重視
- 「演出が使用継続の動力になる」タイプ。軽いテーマ性は歓迎（読みやすさ優先）

---

## 正確性ポリシー（最重要）

Claude Code の機能仕様は**バージョンで頻繁に変わり**、コミュニティ記事には「もっともらしいが実在しない仕様」が混ざる（このハンドブック自身が警告する [[ハルシネーション]] そのもの）。だから本書は次を守る。

- **確実に実在する核心を主軸にする**: `PreToolUse`/`PostToolUse` 等の主要フック、SKILL.md の `name`/`description`、`.claude/agents/`、`claude mcp add`、settings.json の `permissions`（allow/deny/ask）など、公式ドキュメントで確認できるものを中心に据える
- **流動的・高度な仕様は断定しない**: 不確かなフィールドや新機能は「公式ドキュメントで要確認」と明示する。推測で「ある」と書かない（[[推測禁止ルール]]）
- **各章末に公式ドキュメントへのリンクを必ず置く**: 読者が一次情報で裏を取れるようにする
- **コピペで動く最小例は、書いた人（Claude）が構文を確認できる範囲で**書く。確認できない構文は例にしない

## 運用ルール

### 1. 各章の README.md 構成（実用フォーマット）

各章の `README.md` は以下のセクションを持つ。**「重要な単語まとめ」は必ず章末**。

1. `# 第N章: タイトル`
2. `> Status: 執筆予定 / 執筆中 / 完了`
3. `## この章で学ぶこと`
4. `## 前提`
5. `## 本文` — **仕組みは最小限 → 手順 → コピペで動く最小例 → 実用パターン → Windows/PowerShell 注意** の順で実用に寄せる
6. `## つまずきポイント`
7. `## 公式ドキュメント` — 一次情報へのリンク（必須）
8. `## 次の章へ`
9. `## 重要な単語まとめ` ← **章の最後に必ず置く**

### 2. 進捗が更新されたときの作業

章のステータスが変わったら、**3 箇所を同期** する。

- このファイル（`CLAUDE.md`）の「現在の進捗」
- ルート `README.md` の進捗チェックボックス
- 該当章の `README.md` 冒頭の `> Status: ...` 行

### 3. 進め方の方針

- **推測禁止**: 不明点はユーザーに質問する。「たぶんこうだろう」で進めない
- **過剰実装しない**: 1 章 1 テーマ。盛りすぎない
- **必ず「コピペで動く最小例」を入れる**: 読者がその場で試せること。Windows で動く形を優先
- **新規の専門用語には身近な実例を添える**: ユーザーが日常的に触れるもの（自分が作った MAGI・/zenn-log・/save・Notion/Slack/Gmail の MCP など）を引き合いに出す。`## 重要な単語まとめ` の表にも「身近な実例」列を含める
- **Windows/PowerShell 固有の注意を必ず添える**: UTF-8（BOM なし）保存、パスエスケープ、`Out-File` のエンコーディングなど、実際にハマる点を書く
- **自作ツールを実例に**: 第4〜7章は、ユーザーが既に作ったツール（/zenn-log、MAGI、各 skill、MCP 連携）を題材に「なぜこの形にしたか」を言語化する
- **重要単語・要注意点はハイライト**: `> [!CAUTION]` は 1 章あたり 2〜4 個、`<mark>...</mark>` は 5〜8 個が目安。やりすぎない

## 質問ログと番外編（補強ページ）

学習中にユーザーが「わからない」を聞いてきたら、回答と同じターンで `extras/qa-log.md` に 1 エントリ追記する。同じ関連章で「補強済み: ☐」が 3 件たまったら、回答の末尾で「○○について 3 件たまりました。補強ページ作りますか？」と一度だけ提案する。

承認されたら `extras/supplements/NNN-タイトル.md` を作成し、該当エントリを「補強済み: ☑」に更新、`extras/supplements/README.md` の一覧にも追加する。

記録フォーマット:

```
### YYYY-MM-DD: <質問の見出し>
- 関連章: 第N章
- 質問: <要旨 1 行>
- 回答要点:
  - <要点 1>
  - <要点 2>
  - <要点 3>
- 補強済み: ☐
```

**記録する**: 学習目的の質問（「○○とは？」「○○と○○の違いは？」）
**記録しない**: 作業指示（「第3章書いて」）、運用相談、雑談
迷ったら記録する側に倒す。
